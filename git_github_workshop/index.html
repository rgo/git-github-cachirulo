Arrancamos con Git

Configurando Git

Definiendo nuestro usuario e email:

    $ git config --global user.name "Rafa G."
    $ git config --global user.email foo@example.com

Opcional:

Definiendo nuestro editor preferido:

    $ git config --global core.editor joe

Definiendo nuestra herramienta de diff:

    $ git config --global merge.tool meld

Mensajes de salida coloreados:

    $ git config --global color.ui auto

La ayuda de Git:

3 maneras de pedir la ayuda de config:

    $ git help config
    $ git config --help
    $ man git-config

Si no siempre podemos preguntarle a Google :)

Creando un repositorio

Inicializamos un repositorio local:

    $ cd cachirulo/
    $ git init
    Initialized empty Git repository in /home/rafa/cachirulo/.git/

Estado del repositorio

Comprobamos el estado del directorio de trabajo:

    $ git status

Podemos ver su histórico:

    $ git log

Con una interfaz gráfica:

    $ gitg

Nota: El comando "gitg" puede no existir, prueba con "gitk".

Add & commit <3

Añadimos los ficheros y los "commiteamos":

    $ vi ruby_on_beers.rb
    $ git add .
    $ git commit -m "First commit"
    [master (root-commit) 2f50bf2] First commit
     1 files changed, 3 insertions(+), 0 deletions(-)
     create mode 100644 ruby_on_beers.rb

Nota: Haz "git status" en cada paso para ver el estado del fichero.

Git status al detalle

Viendo que ha cambiado en nuestro repositorio y como se representan los cambios:

    $ vi ruby_on_beers.rb   # Realizo cambios
    $ touch README
    $ touch ignore_me
    $ git add README
    $ git status

Git status al detalle

La salida de git status es la siguiente:

Ignorar ficheros

Podemos ignorar ficheros para mantener limpio nuestro "working tree":

    $ echo "ignore_me
    tmp/*
    tmp/**/* " > .gitignore
    $ git add .gitignore
    $ git commit -m 'Ignore ignore_me and tmp files'

Nota: Haz "git status" antes y después de crear el fichero .gitinore para ver el estado del directorio de trabajo

Deshaciendo cambios

Deshaciendo cambios no añadidos a staging

Restaurar un fichero modificado (¡perdiendo los cambios!):

    $ echo "Dábale arroz a la zorra el abad" > README
    $ git checkout -- README

Nota: Generalmente se puede omitir el "--", indica que el resto de argumentos son directorios y/o ficheros

Deshaciendo cambios añadidos a staging

Restaurar un fichero:

    $ echo "Debo confesar que nací a una edad muy temprana" > README
    $ git  add .
    $ git reset HEAD README
    Unstaged changes after reset:
    M	README

Deshaciendo cambios "committed" sin dejar rastro

Deshaciendo cambios "committed" sin dejar rastro

Borrando un commit directamente:

    $ git reset --hard HEAD^

Borrando los 3 últimos commits:

    $ git reset --hard HEAD~3

Nota: "HEAD^" indica revisión anterior de "HEAD", por lo que "HEAD~3" es equivalente a "HEAD^^^"

Deshaciendo cambios "committed" sin dejar rastro

Deshaciendo commits manteniendo los cambios en staging:

    $ git reset --soft HEAD^

Organizando ficheros

Organizando ficheros

Mover ficheros es tan simple como:

    $ git mv ruby_on_beers.rb ruby-on-beers.rb

Borrar ficheros sigue la misma convención:

    $ git rm ruby-on-beers.rb

Nota: Recuerda que debes hacer commit para que los cambios se hagan permanentes.

Ramas

Crear ramas

Crear una rama y moverme a ella:

    $ git branch foo
    $ git checkout foo
    Switched to branch 'foo'

Nota: La misma acción con un solo comando sería "git checkout -b bar".

Listar ramas

Listado de las ramas disponibles:

    $ git branch
      bar
    * foo
      master

Nota: El asterisco indica la rama en la que me encuentro actualmente.

Merge de ramas

"Mergeando" ramas modificadas:

Rebase de ramas

Haciendo "rebase" de las ramas:

    $ git checkout master
    $ git rebase rebase-foo
    First, rewinding head to replay your work on top of it...
    Fast-forwarded master to rebase-foo.
    $ git rebase rebase-bar
    First, rewinding head to replay your work on top of it...
    Applying: rebase foo

Borrando ramas

Para borrar ramas:

    $ git branch -d bar
    Deleted branch bar (was 7fc22ab).

Nota: Si la rama no ha sido "mergeada" hay que usar el parámetro "-D".

Flujo de trabajo normal

  1. Actualizo mi repositorio local con el remoto: 
        $ git pull --rebase
  2. Creo una rama: 
        $ git checkout -b killer-feature
  3. Hago los cambios pertinentes(add & commit) en mi repositorio local
  4. Realizo un rebase: 
        $ git checkout master
    $ git rebase killer-feature
  5. Subo los cambios al repositorio público/remoto: 
        $ git push origin master

Fork repositorios

  1. Pulsamos el botón de fork en el repositorio elegido
  2. Lo clonamos en local: 
        $ git clone git@github.com:username/proyecto-molon.git
  3. Creamos una rama remota apuntando al repositorio original para poder actualizar nuestro fork: 
        $ git remote add upstream git@github.com:owner/proyecto-molon.git
  4. Descargamos repositorio remoto y actualizamos nuestro fork: 
        $ git fetch upstream
    $ git merge upstream/master

    5. Nota: Para actualizar nuestro fork podemos hacer pull o también pull --rebase

Colaborando en proyectos

  1. Hacemos un fork como hemos indicado anteriormente
  2. Creamos una rama: 
        $ git checkout -b fix-world
  3. Arreglamos el mundo... :)
  4. Subimos nuestra rama local a github: 
        $ git push origin fix-world
  5. Desde la interfaz de GitHub nos movemos a la rama "fix-world"
  6. Y pulsamos el botón "Pull Request"

Nota: Información detallada sobre Pull Requests

Nota2: Para borrar la rama remota se utiliza el comando: "git push origin :fix-world"

Git Fatalities

Conflictos

  1. Creamos una rama y modificamos un ficheros
  2. Creamos otra rama y modificamos el mismo fichero
  3. Hacemos rebase de la primera rama
  4. Hacemos rebase de la otra rama y... ¡conflicto!: 
            $ git rebase bar
        First, rewinding head to replay your work on top of it...
        Applying: Lorem ipsum
        Using index info to reconstruct a base tree...
        Falling back to patching base and 3-way merge...
        Auto-merging README
        CONFLICT (content): Merge conflict in README
        Failed to merge in the changes.
        Patch failed at 0001 Lorem ipsum

        When you have resolved this problem run "git rebase --continue".
        If you would prefer to skip this patch, instead run "git rebase --skip".
        To restore the original branch and stop rebasing run "git rebase --abort".
  5. Lo solucionamos y añadimos el fichero
  6. Continuamos con el rebase: 
          $ git rebase --continue
      Applying: Lorem ipsum

Nota: Si el conflicto viene por un merge se resuelve el conflicto, se añade y si "commitea".

Nota2: El comando "git mergetool" te mostrará la interfaz gráfica definida para diffs.

Stash

Ocultando cambios(temporalmente) para trabajar en otra cosa:

    $ git status
    # On branch master
    # Changes not staged for commit:
    #   (use "git add ..." to update what will be committed)
    #   (use "git checkout -- ..." to discard changes in working directory)
    #
    #	modified:   README
    #
    no changes added to commit (use "git add" and/or "git commit -a")
    $ git stash
    Saved working directory and index state WIP on master: 3263921 Lorem ipsum
    HEAD is now at 3263921 Lorem ipsum
    $ git status
    # On branch master
    nothing to commit (working directory clean)
    $ git stash pop

Nota: Recomiendo leer "man git-stash" para ver todas sus opciones.

Blame

Para saber quien es el culpable de un cambio:

    $ git blame README
    $ git blame -L 5 README       # Blame de la línea 5
    $ git blame -L 5,10 README    # Blame de las líneas 5 a la 10
    $ git blame -L 5,+5 README    # Blame de las líneas 5 a la 10

Nota: El parámetro "-L" acepta expresiones regulares como inicio.

Tags remotos

Crear tags en el servidor público:

    $ git tag v2.0
    $ git push --tags

Nota: El push por configuración se hará a origin, explicitamente sería "git push --tags origin".

Eliminar tags en el servidor público:

    $ git push origin :v2.0

Nota: No recomiendo eliminar tags(ni nada) que ya está subido al servidor público.

Un buen flujo de trabajo en equipo

  1. Una rama estable
  2. Una rama master (inestable, de desarrollo)
  3. Crea tu rama local siempre desde estable
  4. Cuando has terminado: "mergea" la rama en master y deploy a staging
  5. Cuando está aceptado: "mergea" la rama en estable y deploy a producción
  6. NUNCA: rebase o merge de master

Bisect

Biseccionar para encontrar un problema:

     $ git bisect start
     $ git bisect bad [commit/branch/tag/etc]
     $ git bisect good [commit/branch/tag/etc]
     $ git bisect run [script para automatizar la prueba]
